# Multi-Agent Refactor Wave 7 Live3D Artifact Integration Plan

Date: 2026-06-29

## Objective

Connect `apps/live3d` to the YOLO and stereo calibration artifact packages
produced by the independent tools:

```text
tools/yolo package -> artifacts/models/tennis_ball_yolo
tools/calibration gui stereo -> artifacts/calibration/stereo_cam1_cam2
apps/live3d fetches JSON -> packages/core validates/converts -> status UI
```

This wave must not add real USB camera capture, YOLO inference, or live 3D
tracking. It should keep fixture rendering for the visual scene while adding a
real artifact-loading boundary.

## Branch

```text
refactor/live3d-artifact-loader
```

## Required Implementation

Add an app-local artifact adapter under `apps/live3d/src`, for example:

```text
apps/live3d/src/artifacts.ts
```

The adapter should:

- define a small JSON reader interface;
- provide a browser `fetch` reader;
- load YOLO package JSON files:
  - `package.json`
  - `labels.json`
  - `preprocessing.json`
  - `postprocessing.json`
- load stereo calibration JSON files:
  - `package.json`
  - `cam1.json`
  - `cam2.json`
  - `stereo.json`
  - `rectification.json`
  - optional `verification.json`
- pass parsed objects into Wave 4 core helpers:
  - `loadYoloModelArtifactMetadata`
  - `loadStereoCalibrationArtifact`
- return explicit success/failure status objects rather than throwing for
  ordinary missing or invalid artifact data.

Update the Live3D UI to render artifact status:

- YOLO package loaded: selected model, model path, confidence threshold, pending
  file/SHA/byte checks;
- YOLO package blocked: missing file or validation error;
- stereo calibration loaded: left/right camera IDs and baseline;
- stereo calibration blocked: missing file or validation error;
- fixture scene still clearly labelled as fixture-only.

Update `apps/live3d/scripts/serve.js` so local development can serve ignored
repo-root artifacts through `/artifacts/...` without copying them into `dist/`.
This should be read-only static serving and must avoid path traversal.

## Config

Prefer browser-facing artifact paths in `src/config.ts`:

```text
/artifacts/models/tennis_ball_yolo
/artifacts/calibration/stereo_cam1_cam2
```

Do not import `tools/yolo` or `tools/calibration` from the app.

## Tests

Add focused Bun tests for:

- successful artifact load from an in-memory JSON reader using fixtures that
  match the packages generated by Wave 5 and Wave 6;
- missing YOLO `labels.json` returns a blocked status with a useful message;
- invalid YOLO class `0` returns a blocked status;
- missing optional calibration `verification.json` produces a warning but can
  still load when package quality is accepted;
- rejected calibration package returns a blocked status;
- `serve.js` path mapping serves `/artifacts/...` only from repo-root
  `artifacts/` and rejects traversal.

## Acceptance Criteria

- `apps/live3d` validates YOLO and stereo calibration artifact JSON through
  `packages/core`, not through tool internals.
- Missing or invalid artifact data is visible in the status panel.
- The app can still build and run without real artifacts, showing blocked
  artifact status plus fixture-only scene rendering.
- The dev server can expose repo-root `artifacts/` through `/artifacts/`.
- No edits occur outside `apps/live3d/**` and `docs/**`.
- Existing `TennisBallDetectorLab` dirty state remains untouched.

## Lead Review Notes

The lead should check:

- no `tools/yolo` or `tools/calibration` imports;
- browser fetch paths are artifact-package roots, not training directories;
- artifact load failures are explicit and user-readable;
- fixture rendering is still not presented as real camera/YOLO validation;
- dev server static file mapping cannot escape the intended directories.
